A Brief History

A work in process limited pull system for managing your workflow.


  • Translates roughly to "signboard" or "card"
  • Developed by Toyota
  • Eventually adapted to software development

  • Developed by Toyota in late 1940s for just-in-time lean manufacturing
  • Kanban named for the physical cards representing work

The Kanban Board

Physical vs. Digital

  • Physical board is preferred, best insight to all team members what the work state is
  • Use a digital board if space constrained, remote employees, just think its really cool to have a digital board and everyone wants one, etc.
  • Automated metric reporting

WIP Limits

Do exactly what the name would suggest...limit the amount of work in process

  • Reduce context switching
  • When properly balanced, decrease lead time, increase throughput
  • Quickly adapt to changing priorities with minimal waste

Selecting WIP Limits

Your process is only as fast as its slowest step.

  • You can only be as fast as your slowest step. Start there.
  • # of people working at that stage * 1.5.
  • Match other stages throughput to that stage.
  • Dev can finish ~1 item per week, per person. 4 people. Set WIP = 6, throughput is ~4 items per week.
  • Test can finish ~2 items per week, per person. WIP = (4 / 2) * 1.5 = WIP = 3.

Using It

Remember, it's a pull system

  • Cards are pulled into the next stage
  • Columns should be prioritized top to bottom
Prioritizing columns is ongoing, and can be double checked at a daily standup. This removes the need to have unnecessary prioritization meetings.

Using It

Set rules about when a card can be considered "done" in a particular stage

Helps drive quality through every step of the process by ensuring things aren't skipped over, like code reviews.

Managing External Dependencies

Set them aside and wait.
If something in progress is blocked on an external dependency, you can just flag it as an external dependency. It doesn't count towards your WIP limits, just revisit each day at your standup and see if it's ready to be worked on again.

Releasing

Kanban is about continuously delivering value to customers rather than dropping a large feature set.
Select a release cadence, or release when you have "enough" ready to go. Don't wait until you have a whole new version with tons of new features to go out. Roll out those features incrementally as they become ready, gather feedback.

Measuring Success

  • Leadtime
  • Throughput
  • New bugs
Your process and your WIP limits can and should change over time to try to drive improved metrics.

TLDR;

Kanban can:
  • Reduce # of planning meetings (especially if you're coming from Scrum)
  • Visualize your workflow and identify bottlenecks
  • Respond faster to changing priorties
  • Continuously deliver value
